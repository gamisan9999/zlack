# zlack デバッグ記録 2026-03-22

## 概要

MVP 実装完了後の E2E テストで複数のランタイムエラーが発生。
Zig 0.15.2 の `std.http.Client` API と `std.json` のメモリ所有権に起因する問題が中心。

## 発見した問題と修正

### 1. stdin 読み取りブロック
- **症状**: トークン入力後に何も起きない
- **原因**: `stdin.allocRemaining()` が EOF まで読もうとし、ターミナルではブロック
- **修正**: `readLine()` ヘルパー（1バイトずつ読んで `\n` で停止）に置換
- **コミット**: `71051f0`

### 2. HTTP POST で sendBodiless panic
- **症状**: `thread panic: reached unreachable code` at `sendBodilessUnflushed`
- **原因**: `std.http.Client.fetch()` は `payload=null` のとき `sendBodiless()` を呼ぶが、POST メソッドでは `requestHasBody()=true` のため assert で panic
- **修正**: 空パラメータでも `""` を payload として渡す
- **コミット**: `29cf18e`

### 3. HTTP レスポンスボディが空
- **症状**: `JsonParseFailed` — HTTP 200 だがボディが空
- **原因**: `sendBodyUnflushed(&.{})` に空スライスを渡すと HTTP ヘッドが正しくフラッシュされない
- **修正**: `fetch()` API に全面移行。レスポンスは `Writer.fixed()` で受信
- **コミット**: `e833e43`

### 4. fetch のレスポンスバッファオーバーフロー
- **症状**: `WriteFailed` for `users.list`
- **原因**: `Writer.fixed()` の 64KB スタックバッファでは大きなレスポンスに不足
- **修正**: 1MB ヒープアロケートバッファに変更
- **コミット**: `4eb66f2`

### 5. use-after-free (authTest)
- **症状**: Segfault at `allocator.dupe(u8, auth_resp.team_id ...)`
- **原因**: `apiCallParsed` 内で `defer self.allocator.free(body)` → `body` は JSON パーサの入力ソース。`Parsed.value` の文字列フィールドは `body` のスライスを直接参照。`defer free(body)` で dangling
- **修正**: `authTest` で `deinit` 前に全文字列を `dupe`。`apiCallParsed` は body を返すように変更
- **コミット**: `7be8d00`

### 6. use-after-free (全 API メソッド)
- **症状**: Segfault at `pagination.zig:33 appendSlice`
- **原因**: `conversationsList` の fetch コールバック内で `defer parsed.deinit()` → `return parsed.value` で channels スライスが dangling。`@memcpy` もシャローコピーなので文字列ポインタは無効
- **修正**: 全 API メソッドで `body` と `parsed` の deinit/free を削除（MVP ではメモリ leak 許容）。pagination コールバックは単一ページ fetch に簡素化
- **コミット**: `4ee6685`

### 7. conversations.list の scope エラー
- **症状**: `missing_scope: mpim:read`
- **原因**: `types=public_channel,private_channel,mpim,im` で mpim/im の scope がない
- **修正**: `types=public_channel,private_channel` に変更（DM は Phase 2）
- **コミット**: `4eb66f2`

## 現在のステータス

最後のテスト実行ログ（環境変数トークン使用時）:

```
Step 1: Auth OK
Step 2: Init Slack client
Step 3: Init database
Step 4: TUI OK
Step 5: Fetch channels → HTTP 200
Step 5: Fetch users
Step 6: Socket Mode → HTTP 200
Step 7: Enter main loop
```

Step 7（メインイベントループ）まで到達。ただしユーザーの最終テストでは古いバイナリが使われた可能性あり（`zig build` 後に `./zig-out/bin/zlack` を実行する必要がある。`zig build run` なら自動ビルド込み）。

## 根本原因の考察

Zig の `std.json.parseFromSlice` は入力スライスの文字列を直接参照する（コピーしない）。
`Parsed(T).deinit()` はアリーナを解放するが、入力バッファは解放しない。
したがって `Parsed.value` を使う間は:
1. 入力バッファ（`body`）を生かす
2. `Parsed` アリーナを生かす（`deinit` しない）

の両方が必要。MVP では leak を許容して回避。

## 今後の改善案

1. **ArenaAllocator パターン**: アプリ全体で 1 つの ArenaAllocator を使い、終了時にまとめて解放
2. **ディープコピー関数**: `dupeChannel()`, `dupeMessage()` 等を実装し、全フィールドを owned にする
3. **ページネーション復活**: ディープコピー対応後に pagination.zig を再統合
4. **環境変数/Keychain の統合**: team_id ベースの Keychain 検索を改善

## 環境変数での実行方法

```bash
ZLACK_USER_TOKEN=xoxp-... ZLACK_APP_TOKEN=xapp-... zig build run 2>/tmp/zlack-debug.log
```

または事前ビルド済みの場合:
```bash

```
